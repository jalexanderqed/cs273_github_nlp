   public interface RowVisitor {
        public boolean processRow(byte[] key, byte[] data, boolean isFirst, boolean hasMore);
    }


   public void forEach(String operationName, byte[] startKey, boolean stopOnKeyChange, RowVisitor visitor, boolean keysOnly, boolean fillCache) {
         RocksIterator iterator = null;
        try {
            iterator = _database.newIterator(_dataColumnHandle);

            if (startKey == null)
                iterator.seekToFirst();
            else
                iterator.seek(startKey);


           boolean first = true;
            for (; iterator.isValid(); iterator.next()) {
                if (iterator.isValid()) {
                    byte[] rowKey = iterator.key();

                    if (stopOnKeyChange && startKey != null)
                        if (rowKey.length < startKey.length || !ArrayUtils.isEquals(startKey, ArrayUtils.subarray(rowKey, 0, startKey.length))) {
                            //end of seek set
                            break;
                        }
                    byte[] data = keysOnly ? null : iterator.value();

                    try {
                        if (!visitor.processRow(rowKey, data, first, iterator.isValid())) break;
                    } catch (Exception e) {
                        Log.error("Error processing row!", e);
                    }
                    first = false;
                }
        } finally {
            try {
                if (iterator != null) iterator.dispose();
            } catch (Throwable e) {
                Log.error("Unable to close the iterator, could result in a memory leak!", e);
            }
        }

The code that causes this was original written with a LevelDB implementation (leveldbjni 1.8), and this code runs fine.  Going over the details it seems a difference is that the leveldbjni library did not attempt to cleanup objects with a finalize method and you had to explicitly call dispose() or leak memory.

My theory (unconfirmed but testing) is that the JVM is calling finalize() while my code is calling dispose(), leading to a possible race condition.  This seems to be more evident when using CMS, as well as on a busy server.  My initial reaction as that this would not be possible because the dispose() method is synchronized, however I have read some articles that imply that it may be possible that the finalize() is in fact called.
